● ClickHouse 跳数索引 (Skip Index) 介绍

  什么是跳数索引

  跳数索引是一种数据跳过机制，在数据块级别（如每8192行）预先计算并存储该块内数据
  的统计信息，查询时快速判断该数据块是否可能包含目标值，从而跳过不相关的数据块。

  工作原理

  原始数据 (8192行/块) →  跳数索引计算  →  索引存储统计信息
                                      ↓
  查询时: 根据索引判断数据块是否可能匹配 → 跳过不匹配的块 → 只扫描相关块

  支持的索引类型

  ┌───────────────┬───────────────────┬─────────────────────────────┐
  │   索引类型    │     存储内容      │          适用场景           │
  ├───────────────┼───────────────────┼─────────────────────────────┤
  │ minmax        │ 块内最大值/最小值 │ WHERE price > 100           │
  ├───────────────┼───────────────────┼─────────────────────────────┤
  │ set(max_rows) │ 块内所有唯一值    │ 低基数列 WHERE status = 'A' │
  ├───────────────┼───────────────────┼─────────────────────────────┤
  │ Bloom filter  │ 布隆过滤器        │ WHERE name LIKE '%张%'      │
  ├───────────────┼───────────────────┼─────────────────────────────┤
  │ tokenbf_v1    │ token布隆过滤器   │ 字符串包含查询              │
  ├───────────────┼───────────────────┼─────────────────────────────┤
  │ ngrambf_v1    │ ngram布隆过滤器   │ 模糊匹配、前缀搜索          │
  ├───────────────┼───────────────────┼─────────────────────────────┤
  │ probability   │ 概率密度函数      │ 范围查询优化                │
  └───────────────┴───────────────────┴─────────────────────────────┘

  你的表结构适合添加哪种

  根据 stock_daily 表的查询模式：

  ┌─────────────────────────────────────────┬────────────┬──────────────────┐
  │                查询场景                 │  推荐索引  │       说明       │
  ├─────────────────────────────────────────┼────────────┼──────────────────┤
  │ WHERE stock_code = '000001'             │ minmax 或  │ 股票代码基数中等 │
  │                                         │ set        │                  │

  │ (从stock_info关联)                      │            │                  │

  如何添加跳数索引

  -- 添加跳数索引
  ALTER TABLE stock_scraper.stock_daily ADD INDEX idx_stock_code stock_code TYPE
   minmax;
  ALTER TABLE stock_scraper.stock_daily ADD INDEX idx_trade_date trade_date TYPE
   minmax;
  ALTER TABLE stock_scraper.stock_daily ADD INDEX idx_change_pct change_pct TYPE
   minmax;

  -- 为现有数据生成索引
  ALTER TABLE stock_scraper.stock_daily MATERIALIZE INDEX idx_stock_code;
  ALTER TABLE stock_scraper.stock_daily MATERIALIZE INDEX idx_trade_date;
  ALTER TABLE stock_scraper.stock_daily MATERIALIZE INDEX idx_change_pct;

  注意事项

  1. 写入性能: 索引会在写入时计算，轻微增加CPU开销
  2. 存储开销: 索引数据通常很小（约1-5%额外空间）
  3. MATERIALIZE: 索引添加后需要为历史数据触发索引计算
  4. 局限性: 对于需要扫描全表的聚合查询（如 COUNT(*)）帮助有限

  ClickHouse 主键索引机制

  你的误解

  主键 (stock_code, trade_date) 不是传统意义的联合索引，ClickHouse使用的是稀疏索
  引，这与MySQL/InnoDB等完全不同。

  ClickHouse 稀疏索引原理

  MySQL (密集索引):         ClickHouse (稀疏索引):
  每行都有索引条目          每8192行一个索引条目
  ├───├───├───├───├───    ├───┬───┬───┬───┬───┬───┬───
  │ 1 │ 2 │ 3 │ 4 │ 5 │    │ 1 │ 9 │17 │25 │33 │41 │ 49
  └───┴───┴───┴───┴───┘    └───┴───┴───┴───┴───┴───┴───
                           数据块1  数据块2  数据块3 ...

  关键点：数据按主键排序，但不分区

  你的数据按 (stock_code, trade_date) 排序，意味着：
  - 先按股票代码排序
  - 同股票内按日期排序

  但问题是：数据块（Granule）大小是 ~8192行，一个数据块可能包含：
  - 一只股票的一天数据 × 8192天
  - 或者多只股票的数据

  为什么 WHERE stock_code = '000001' 仍然慢

  假设 000001 平安银行有 2500个交易日 的数据：

  数据块1: [000001-2020-01-02, 000001-2020-01-03, ..., 000002-2020-01-02, ...]
  (8192行)
  数据块2: [..., 000005-2025-12-30, 000005-2025-12-31, 000006-2020-01-02, ...]
  (8192行)
  ...

  ClickHouse需要扫描所有数据块，因为：
  1. 索引只记录每个数据块的边界值（第一行和最后一行）
  2. 无法快速判断某个数据块是否包含 stock_code = '000001'
  3. 必须逐块读取、判断、再跳过

  主键适合的场景

  ┌────────────────────────────────────────────────┬────────────────────────┐
  │                    查询类型                    │      主键索引效果      │
  ├────────────────────────────────────────────────┼────────────────────────┤
  │ WHERE stock_code = 'xxx' AND trade_date =      │ ✅ 极速（精确到一行）  │
  │ 'yyy'                                          │                        │
  ├────────────────────────────────────────────────┼────────────────────────┤
  │ WHERE stock_code = 'xxx' AND trade_date        │ ✅                     │
  │ BETWEEN ...                                    │ 好（同股票连续数据）   │
  ├────────────────────────────────────────────────┼────────────────────────┤
  │ WHERE stock_code = 'xxx'                       │ ⚠️ 一般（需扫描多块）  │
  ├────────────────────────────────────────────────┼────────────────────────┤
  │ WHERE trade_date BETWEEN ...                   │ ❌                     │
  │                                                │ 差（跨股票数据混合）   │
  └────────────────────────────────────────────────┴────────────────────────┘

  解决方案：添加跳数索引

  跳数索引会在每个数据块内计算统计信息：

  -- 添加后，查询 stock_code 会直接跳过不包含该股票的数据块
  ALTER TABLE stock_scraper.stock_daily ADD INDEX idx_stock_code stock_code TYPE
   minmax;
  ALTER TABLE stock_scraper.stock_daily MATERIALIZE INDEX idx_stock_code;

  效果对比：

  无跳数索引:  扫描所有数据块 → 读取13M行 → 过滤
  有跳数索引:  快速跳过不含该股票的数据块 → 读取~5000行 → 完成

  总结

  ┌───────────────────────────────┬────────────────┬───────────────────────┐
  │           索引类型            │    作用位置    │     你的表适合度      │
  ├───────────────────────────────┼────────────────┼───────────────────────┤
  │ 主键 (stock_code, trade_date) │ 数据块边界     │ ✅ 范围查询有效       │
  ├───────────────────────────────┼────────────────┼───────────────────────┤
  │ 跳数索引 minmax               │ 数据块内部统计 │ ✅ 点查优化必选       │
  ├───────────────────────────────┼────────────────┼───────────────────────┤
  │ 分区 (trade_date)             │ 月/日分区裁剪  │ ⚠️ 按日期范围查询有效 │
  └───────────────────────────────┴────────────────┴───────────────────────┘

  结论：主键索引不等于传统数据库的联合索引，ClickHouse需要跳数索引来高效支持单列
  点查。

  三、物化视图优化 (针对聚合查询)

  对于 GROUP BY 聚合查询，跳数索引帮助有限，建议创建物化视图：

  3.1 年度统计物化视图

  -- 创建年度统计物化视图 (查询耗时从600ms降到<10ms)
  CREATE MATERIALIZED VIEW stock_daily_yearly_stats
  ENGINE = SummingMergeTree()
  ORDER BY (year, stock_code)
  AS
  SELECT
      toYear(trade_date) AS year,
      stock_code,
      COUNT(*) AS trade_count,
      SUM(volume) AS total_volume,
      SUM(turnover) AS total_turnover,
      AVG(close) AS avg_close,
      MAX(high) AS max_high,
      MIN(low) AS min_low,
      MAX(change_pct) AS max_change_pct,
      MIN(change_pct) AS min_change_pct
  FROM stock_scraper.stock_daily
  GROUP BY toYear(trade_date), stock_code;

  原查询:
  -- 耗时 ~600ms
  SELECT toYear(trade_date), COUNT(*), COUNT(DISTINCT stock_code)
  FROM stock_daily GROUP BY toYear(trade_date)

  优化后:
  -- 耗时 <10ms
  SELECT year, COUNT(*), uniqExact(stock_code)
  FROM stock_daily_yearly_stats GROUP BY year

  3.2 月度统计物化视图

  CREATE MATERIALIZED VIEW stock_daily_monthly_stats
  ENGINE = SummingMergeTree()
  ORDER BY (year, month, stock_code)
  AS
  SELECT
      toYear(trade_date) AS year,
      toMonth(trade_date) AS month,
      stock_code,
      COUNT(*) AS trade_count,
      SUM(volume) AS total_volume,
      AVG(turnover_rate) AS avg_turnover_rate
  FROM stock_scraper.stock_daily
  GROUP BY toYear(trade_date), toMonth(trade_date), stock_code;

  四、分区策略优化

  当前数据按 ReplacingMergeTree 无分区，建议按月分区：

  -- 重建表为按月分区

  -- 迁移数据
  INSERT INTO stock_scraper.stock_daily
  SELECT * FROM stock_scraper.stock_daily_old;

  DROP TABLE stock_scraper.stock_daily_old;

  - 按月分区后，跨月查询可跳过无关分区
