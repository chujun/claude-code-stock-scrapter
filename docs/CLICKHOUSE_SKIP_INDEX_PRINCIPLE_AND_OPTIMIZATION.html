跳数索引工作流程

-- 添加跳数索引 ALTER TABLE stock_daily
  ADD INDEX idx_stock_code stock_code TYPE minmax;

-- 为历史数据生成索引 ALTER TABLE stock_daily
  MATERIALIZE INDEX idx_stock_code;
1. 查询 stock_code='000001' ←
2. 检查 idx_stock_code ←
3. 对每个 Granule：min ≤ 000001 ≤ max ? ←
4. 只扫描匹配的数据块 ✓

ClickHouse vs MySQL 索引对比

特性 MySQL (InnoDB) ClickHouse
索引类型 密集索引 (每行都有) 稀疏索引 (每8192行)
索引结构 B+Tree 稀疏排列 + 跳数索引
主键作用 聚簇索引，存储行数据 数据按主键排序存储
优化机制 ICP 下推 数据跳过 (Skip)
设计目标 OLTP · 事务性 OLAP · 分析型

查询场景性能对比

查询场景 MySQL CH主键 CH+跳数索引
WHERE a=1 AND b=2 ✅ O(log n) ✅ O(log n) ✅ O(log n)
WHERE a=1 ✅ O(log n) ⚠️ O(√n) ✅ O(log n)
WHERE b=2 ❌ 全表 ❌ 全表 ❌ 需跳数
GROUP BY a ✅ O(n log n) ⚠️ O(n) ✅ O(n)

ClickHouse 主键索引是"粗略导航"，需要配合跳数索引才能精确定位！

最佳实践

✅ 推荐

  • 为高频查询列添加跳数索引
  • 为低基数列添加 set 索引
  • 定期查看索引使用情况
  • 物化视图优化聚合查询

❌ 避免

  • 为主键已覆盖的组合查询加冗余索引
  • 为几乎所有列都建索引
  • 认为加了索引就能加速
  • 跳过 MATERIALIZE 步骤

经验法则：优先为高频查询列建索引，3-5个索引足够应对大多数场景
